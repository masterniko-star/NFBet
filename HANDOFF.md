## 1. Что это

**NF PlayOff** — веб-приложение для дружеского тотализатора на плей-офф ЧМ-2026.
Паримутуэль (parimutuel): все ставки на матч идут в одну кассу, угадавшие исход делят кассу
пропорционально размеру ставки. Реальные деньги 1:1 (вход 100 ₪, ставки 1–10 ₪, баланс = деньги).
Плей-офф = два исхода (1/2). Лиговые матчи допускают ничью (X).
**Турнир стартует 28 июня 2026.** Участники уже тестируют.

Язык интерфейса — иврит, **RTL**.

### Стек (важно — нестандартный)
- **Клиент** — ОДИН файл `index.html` (~198 КБ). Это и исходник, который ты редактируешь,
  и то, что деплоится (Netlify отдаёт `index.html`). Весь JS — внутри одного `<script>`.
  *(В прошлых сессиях файл назывался `current.html` — это тот же самый файл.)*
- **БД** — Firebase Realtime Database через **REST** (без SDK), база **ОТКРЫТА** (без авторизации).
  URL: `https://nf-playoff-default-rtdb.europe-west1.firebasedatabase.app`.
  ⚠️ **Dev и prod используют ОДНУ И ТУ ЖЕ базу.**
- **Сервер** — одна Netlify scheduled function `netlify/functions/check-results.mjs`,
  cron `* * * * *` (каждую минуту). **Крон работает ТОЛЬКО на prod.** Подтягивает результаты,
  начисляет выигрыши, делает авто-загрузку новых матчей, авто-снятие неактивных и т.д.
- **Хостинг** — Netlify.
  - Prod: `https://nf-playoff.netlify.app/`
  - Dev-алиас: `https://playoff-dev--nf-playoff.netlify.app/`
  - **Админка**: к любому URL добавить `#ctrl7`.

## 5. КРИТИЧЕСКИЕ ПРАВИЛА (не нарушать)

- 🔴🔴 **ПРОД-ДЕПЛОЙ — ТОЛЬКО ПО ЯВНОМУ УКАЗАНИЮ NIKO.** `netlify deploy --prod` запускать
  ИСКЛЮЧИТЕЛЬНО когда Niko прямым текстом сказал «сделай прод деплой» (или явный эквивалент).
  **Всегда переспросить и дождаться подтверждения ПЕРЕД самим `--prod`** — даже если указание уже было.
  Никаких прод-деплоев «по смыслу», «заодно» или по молчаливому согласию. Dev-деплой
  (`--alias=playoff-dev`) — можно по обычному согласию, без этой церемонии.
- 🔴 **Не редактировать JS через Python `re`/regex-substitution** — ломает бэктики/экранирование шаблонных
  строк. Только прямые правки / sed / поиск-замена в редакторе. Python `str.replace` для ЧТЕНИЯ — ок.
  **После любой правки — `node --check`.**
- 🔴 **«Лево»/«право» в просьбах Niko = ФИЗИЧЕСКИЕ стороны экрана.** Не переворачивать под RTL.
- 🔴 **Деплой клиента** → `netlify deploy --dir=. --alias=playoff-dev` (dev).
  **Изменения сервера** (`check-results.mjs`) → `netlify deploy --dir=. --prod` (крон только на prod;
  `--prod` отправляет И клиент, И сервер из `--dir=.`).
- 🔴 **calcMatch в клиенте и в сервере должны давать ОДИНАКОВЫЙ результат.** Это проверяет `paritytest`
  и фаззеры (`moneyfuzz` 46k раскладов, `srvfuzz` 16k сверок). Меняешь расчёт — меняй в ОБОИХ местах.
- 🔴 **ЗАКОН ДЕНЕГ.** Любая ставка, включая авто-ставку, ставится ТОЛЬКО игроку, у которого реально
  есть деньги на балансе; в минус — НИКОГДА. Два условия перед любой ставкой (ручной и авто): (1) игрок
  НЕ в демо (`feePaid===true`) — демо-игроки не получают НИКАКИХ ставок, включая авто; (2) доступный
  баланс считается от РЕАЛЬНОГО депозита (`depOf`/`startBal` на клиенте, `srvDep` на сервере), НЕ от
  плоского `bank`. Касается обеих авто-заливок (клиент `autoBetFill`, сервер в `runCheck`) и `overBetSweep`.
  Примечание: игрок с `feePaid===undefined` (бывает только в старых деревьях) трактуется как демо.
- **Тест-закон**: на каждое изменение — полный прогон набора + новый тест под новое поведение.
- **Дизайн-закон**: повторяющиеся UI-элементы выровнены по сетке; максимум площади — основному контенту.
- Никаких `localStorage`/`sessionStorage` (приложение их не использует; состояние — в Firebase).

## 8. Открытые задачи / TODO (решение за Niko)

1. ✅ **РЕШЕНО (версия 2026-06-28n / srv-2026-06-28a).** Округление паримутуэль-выплат.
   `calcMatch` (клиент `index.html` ~526 и сервер `check-results.mjs` ~32) теперь выплачивает в **целых
   שקלים** методом **наибольшего остатка** (Гамильтон): `base_i = floor(pool*stake_i / sw)`, неделимый
   остаток `pool − Σbase` раздаётся по +1 ₪ победителям с наибольшей дробной частью (`pool*stake_i mod sw`),

## 10. Предпочтения Niko (стиль работы)

### Модель и effort
- **Модель:** Opus 4.8 (`claude-opus-4-8`) — дефолт для всего проекта (цена ошибки высокая: реальные ₪,
  «закон денег», паритет `calcMatch` клиент↔сервер). Sonnet 4.6 — только для дешёвой рутины; Fable 5 —
  избыточен; Haiku — нельзя (финансовая логика).
- **Effort** (`output_config.effort`, у 4.8 дефолт высокий — `xhigh`):
  - 🔴 **`max`** — любые правки, затрагивающие **деньги/расчёт**: `calcMatch`, начисление выплат,
    `overBetSweep`/`lowBalanceSweep`, integrity, синхронизация клиент↔сервер. Корректность > стоимости.
  - **`xhigh`** — обычная разработка с логикой: фичи, рефакторинг, тесты/фаззеры.
  - **`low`/`medium`** — рутина: чтение логов, мелкие правки текста `תקנון`/гида, переименования.
  - Правило: **деньги → `max`, код → `xhigh`, рутина → `low/medium`.** На 4.8 не задирать effort
    рефлекторно — выше потолок интеллекта, `high`/`xhigh` обычно достаточно.

- Отвечать **по-русски, кратко**, по делу.
- Подтверждать перед крупными изменениями; **честный пушбэк**, не поддакивать.
- **Субагентов в чат-интерфейсе нет** — Niko иногда пишет «запусти N субагентов»; это значит «сделай весь
  список задач». Просто делай всё сам последовательно, без долгих преамбул.
- Не могу пиксельно верифицировать вёрстку (нет headless-браузера) — визуал помечать «проверить на телефоне».
- Каждая выдача файла клиенту заканчивается строкой деплоя на отдельной строке.
